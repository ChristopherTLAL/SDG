# phd-cold-mail

## What this skill does

Given:

- A student's PhD application materials (RP + CV) in the vault under `01_Student/{学生名}/文书材料/`
- Upstream artifacts from the vault skills `phd_supervisor_match` (`{英文名}_Supervisor_Shortlist_*.xlsx`) and/or `faculty-deep-dive` (`{英文名}_Faculty_Research_*.xlsx`) under `01_Student/{学生名}/个性化材料/`
  - Either or both can be present; the parser handles either-or gracefully
  - If neither exists, the skill falls back to a manual `01_Student/{学生名}/个性化材料/PhD套磁/导师名单.md` markdown table

Produces (per supervisor, TWO file types):

- **`{NNN}_{Name}.json`** — STRUCTURED, canonical source of truth. Contains `supervisor`, `contact`, `availability`, `top_rank`, `research` (summary + papers + links + alignment_notes), `email` (subject + body + signature), `meta`, `send_status`. This is what downstream tooling consumes (auto-send via send-email skill, send-status tracker, batch re-render).
- **`{NNN}_{Name}.md`** — Human-readable rendering of the JSON, for the user to review and copy-paste into Gmail. Generated by `scripts/draft_io.py json-to-md`. Re-running this command regenerates MD from JSON at any time.

Plus:
- `INDEX.md` summarizing the batch
- Vault-closure side effects (Phase 4): updated 学生.md YAML, 待办看板 entry, 日报 entry routed to the right 中期顾问

**Why JSON + MD (not just MD)**: future auto-send (via the `send-email` skill) needs structured access to subject/body/email/papers separately. MD with frontmatter and pasted sections is hard to parse robustly; JSON is trivial. The user reads MD (the rendering); programs read JSON (the source). `scripts/draft_io.py` provides round-trip-safe MD↔JSON conversion.

**No emails are sent.** The user reads each draft, manually copies subject/body, attaches RP+CV PDFs, and sends from their own mail client. (This matches the n8n workflow this replicates — n8n emails the package to `automation.mail985@gmail.com` for manual forward; here we just land files in the vault.)

1. Resolve student name. If the user mentioned a name in the trigger ("给王稼鑫写套磁"), use that. Otherwise ask.
2. Resolve vault root: `echo $OBSIDIAN_VAULT_ROOT` via Bash; if empty, default to `/Users/shijie/Obsidian/规划看板`.

4. **Read 学生档案**: `Read` `01_Student/{学生名}/{学生名}.md`. Capture from YAML:

5. **Find RP**: Look in `01_Student/{学生名}/文书材料/` for files matching `*RP*` / `*research*proposal*` / `*SoP*` / `*statement*purpose*` (case-insensitive). Prefer PDF if multiple. If multiple plausible matches, list them and ask.
6. **Find CV**: same folder, files matching `*CV*` / `*resume*`. Prefer the `.tex` if both `.tex` and `.pdf` exist (cleaner text), else PDF.

   - Stderr: source paths + counts (e.g. `13 shortlist + 0 faculty → 13 unique professors (2 marked as already contacted)`)
   - If the script exits with `ERROR: No upstream Excel found`, fall back to the markdown table at `个性化材料/PhD套磁/导师名单.md`. If THAT is also missing, print the expected schema (see [Shortlist format](#shortlist-format)) and ask the user to provide one.
8. **Confirm with user before running** — print a summary including skip status:
   ```

   RP:   .../文书材料/魏子涵_RP.pdf

     ✓ shortlist (phd_supervisor_match): 13 位

     2. Dr. Maria Perez-Ortiz — UCL CS

     - Prof. Daniel Alexander — UCL CS (套磁过了)

   输出: .../个性化材料/PhD套磁/邮件草稿/
   ```
   Wait for explicit "go" before kicking off Phase 1. Also accept directives like "去掉 #3 和 #7"、"也跑已联系那两位" — adjust the working list accordingly.

### Phase 1 — Per-supervisor deep research (parallel Sonnet subagents)

In a **single message**, emit one `Agent` tool call per supervisor — they run concurrently. **Max 3 in flight at once**; batch if there are more than 3 supervisors.

Each `Agent` call uses:

- `subagent_type: "general-purpose"`
- `model: "sonnet"`
- `description: "PhD research · <supervisor name>"`
- `prompt`: load [`references/research_prompt.md`](references/research_prompt.md), substitute the shortlist row's fields.

**Choose the right substitution mode based on what upstream data is available** (the parser tells you this via the `source` field on each row):

| `source` value | Means | What to ask subagent |
|---|---|---|
| `shortlist` | Only `phd_supervisor_match` data — name + school + dept + match_reason + homepage | FULL research: profile + papers + email + recruiting status (high WebFetch budget) |
| `faculty` | Only `faculty-deep-dive` data — background + papers already done | LIGHT research: pass papers + background through as preliminary, focus on email + current recruiting status (low WebFetch budget) |
| `shortlist+faculty` | Both upstreams covered this prof | LIGHT research (same as faculty-only) |

This avoids re-doing the paper research that `faculty-deep-dive` already did with its 双盲 validation — save ~70% of API calls in the common case where the user ran both upstream skills.

The subagent's task: WebSearch + WebFetch to confirm the right person + their official pages, then write a structured Markdown report (≤500 zh chars) covering:

- 总结 (~300 chars on research focus — REUSE `background` from faculty-deep-dive if present)
- 机构/学校
- 博士招生情况 (`招收中` / `不确定` / `已暂停/不招收`)
- 1-3 重要论文 (APA — REUSE `papers` from faculty-deep-dive if present; only re-fetch if missing)
- 独特优势 (跟此人读博的 specific upside, 1-2 sentences)
- 邮箱 (must be verifiable — say "未确认" if not certain)
- 重要链接 (≤5, full URLs)

The subagent's return value is the structured Markdown — keep it in memory keyed by supervisor name.

**Rate limit handling**: if a subagent fails or hits rate limits, retry once with a fresh Agent call. If still failing, mark that supervisor as `research_failed` and continue — do not abort the batch.

### Phase 2 — Cold-mail drafting

**Two modes** — choose based on batch size:

#### Mode A: Opus inline (default for small batches, ≤ 30 supervisors)

The coordinator (you, Opus) writes each mail directly — no subagent needed because:

1. Drafting is single-pass and benefits from the coordinator's full context (RP+CV already in working memory after Phase 0).
2. Avoids context-loss between research collection and drafting.

For each supervisor:

1. Read [`references/email_prompt.md`](references/email_prompt.md) — this is the prompt the n8n `gemini-3.1-pro-preview` node uses, translated faithfully.
2. Mentally substitute:
   - `{PROFESSOR_RESEARCH}` ← the Phase 1 markdown
   - `{STUDENT_RP}` ← extracted RP text
   - `{STUDENT_CV}` ← extracted CV text
3. Draft the email. Strict output format (matches the n8n prompt):

   ```markdown
   Subject: <one-line subject>

   <email body — 4 paragraphs corresponding to the 4 points>

   <student name>
   ```

   No markdown bold/italic, no bullet points, no salutation other than the body opener. Single name as signature — no email/phone/affiliation (n8n prompt is explicit about this).
4. Wrap in a small file header — see [Output format](#output-format).
5. Write to `01_Student/{学生名}/个性化材料/PhD套磁/邮件草稿/{NNN}_{Supervisor Name}.md` (NNN = zero-padded `#` column from the Excel if present, else rank; sanitize name for filesystem: replace `/` with `_`, strip leading `Dr.` / `Prof.` if it makes the filename awkward — but keep it inside the file).

#### Mode B: Sonnet fused subagents (full-batch mode, > 30 supervisors)

When batch size > 30, switch to fused end-to-end subagents — ONE Sonnet subagent does research + drafting + file-write for ONE supervisor. The coordinator just spawns batches and collects 1-line statuses. This avoids:
- Burning the coordinator's context with 132 research blobs + 132 draft bodies
- Hours of sequential Opus drafting in one session
- Risk of session interruption losing in-flight work (each subagent writes its own file)

**Phase 1 and Phase 2 fuse into one subagent**. Skip the Phase 1 step described above.

For each supervisor row, emit one `Agent` call:

- `subagent_type: "general-purpose"`
- `model: "sonnet"`
- `description: "Cold-mail · ★{rank or "-"} · <supervisor name>"`
- `prompt`: load [`references/end_to_end_prompt.md`](references/end_to_end_prompt.md), substitute `{SUPERVISOR_NAME}`, `{KNOWN_SCHOOL}`, `{DEPARTMENT}`, `{MATCH_REASON}`, `{TOP_RANK}`, `{HOMEPAGE}`, `{STUDENT_RP}` (full text inline), `{STUDENT_CV}` (full text inline), `{OUTPUT_PATH}` (absolute path the subagent should write to).

**Batching strategy**:
- 5-7 parallel subagents per message (drop to 2-3 if rate-limited).
- All N subagents in a batch must be in ONE message (multiple Agent tool calls in the same response) to actually run concurrently.
- Wait for batch to complete, collect their 1-line statuses, then send the next batch in the next message.
- Track progress with the Task tools (one task per supervisor or per batch — your choice).

**Per-supervisor cost**: the end-to-end prompt enforces multi-pass reasoning (Phases A-G) to make Sonnet approach Opus quality. Each subagent uses noticeably more tokens than Phase 1+2 split, but skips an entire coordinator-side Opus drafting pass.

**Time budget**: 132 supervisors at 5 parallel ≈ 27 batches × ~4 min/batch ≈ ~2 hours wall clock. Most of that is research WebFetch.

**Failure handling**: if a subagent returns ❌ or ⚠️, log it but continue the batch. After all batches done, list failures in the INDEX so the user can re-run them manually.

### Phase 2.5 — Email verification pass (MANDATORY — do not skip)

**The skill's hard goal is to deliver verified emails, not plausible-looking guesses.** After Mode B drafting completes, run a verification + escalation loop:

1. **Local verification** — Run [`scripts/verify_emails.py DIR`](scripts/verify_emails.py). For each draft JSON, this:
   - Fetches the URLs in `research.links` and grep's for the literal email string. If found → marks `email_source = <URL>` (high confidence, 🟢).
   - If grep fails but the email's domain matches the institution → marks `PATTERN-INFERRED:<domain>` (🟡, MUST be escalated).
   - If both fail → marks `email_confirmed = false`, prints to stdout for re-hunt (🔴).

2. **Hunt-confirmed records WITHOUT URL source** — Check JSONs where `contact.email_confirmed=true` and `email_source` is empty or a non-URL string like `directory` / `homepage` from the first hunt pass. These ALSO need verify-or-replace because hunt subagents sometimes forgot to write `email_source`. Run the same verify_emails.py with `--all` over the affected files (or just include them all in step 3).

3. **Escalation: paper-PDF hunt for all 🟡** — Past audit caught ~12% of pattern-inferred emails were actually wrong. So for EVERY 🟡 record:
   - Render a focused hunt prompt with [`references/email_hunt_prompt.md`](references/email_hunt_prompt.md), pre-pended with a "verify-or-replace" context block showing the claimed email and instructing the subagent to either confirm via paper-PDF/homepage fetch OR replace.
   - Spawn 5-7 parallel Sonnet hunt subagents per batch.
   - Each subagent MUST hit the paper-PDF channel (corresponding-author asterisk) before giving up. The hunt prompt enforces this.

4. **Re-verify after hunt** — Re-run `verify_emails.py` to confirm all 🟡 are now 🟢 or marked failed honestly.

5. **Re-render MD** — `python3 scripts/draft_io.py json-to-md DIR` to sync MD files with updated JSON emails.

**Hard rule**: an `email_confirmed=true` row with an empty `email_source` is INVALID. Every confirmed email must point to a URL or paper-PDF identifier.

### Phase 2.55 — Citation & factual verification pass (MANDATORY — do not skip, runs on 100% of drafts)

The single worst error this skill can ship is a draft that asserts a **falsehood the recipient can personally check** — a paper they never wrote, a wrong statute article, a case credited with the wrong doctrine, a fabricated project of theirs, or an inflated title. The recipient is the author / the expert / the named person — they spot it instantly, and it sinks the application. The drafter's own anti-fabrication rules reduce this but do NOT eliminate it; only an independent web re-check does.

**Scope = the WHOLE factual surface, not just citations.** A 2026-06-02 adversarial sweep of 87 drafts found 10 confirmed errors and **6 were not "cited works"**: a fabricated project ("REFLECT-ML"), two wrong statute numbers (Data Act "Art. 4"→43; AI Act "Title VIII"→Chapter V), two wrong case attributions (obtaining-vs-creating credited to "Apis/Innoweb"/"Innoweb" instead of British Horseracing Board/Fixtures Marketing), and two Senior Lecturers called "Professor". Earlier runs also invented paper titles ("Udsen, 2026 introduction to the AI Regulation"; "Lukoseviciene, *Human Agency at the Core* SSRN 2026" — neither existed) and mischaracterized real papers (Buccafusco harms-taxonomy → "hedonic-cost framework"; Geiger → a "metadata paradox" his paper never frames). So verify **cited works, legal cases, statute/regulation numbers + rules, professor titles/affiliations, named projects, and dates.** (See `style_lessons.md` Lessons 16-17.)

> **Proven pattern (use a Workflow for batches > ~20):** `find → adversarial-verify` pipeline. Stage 1 agents extract + web-verify every factual claim per draft; stage 2 *independent* agents re-check only the flags (filter false alarms, confirm real errors). On the 87-draft run this returned 10 confirmed / 0 false-alarm / 0 uncertain. The adversarial second pass matters — it stops both false positives and rubber-stamping.

1. **Local provenance gate (instant, no web)** — `python3 scripts/verify_citations.py DIR`. For every draft it pulls the quoted work-titles out of the body and checks each against that draft's own `research.papers[]` (which must each carry a `source_url`). A title named in the body but absent from the verified papers list = a body-only citation with no provenance = the fabrication signature → BLOCKING. (If the pipeline's drafts don't carry a `papers[]` list, this gate is skipped and web verification below is the sole guard.) The script also splits all drafts into `cite_batch_*.json` for step 2.
2. **Independent web verification (parallel subagents)** — spawn one `general-purpose` subagent per batch using [`references/verify_citations_prompt.md`](references/verify_citations_prompt.md) (substitute `{BATCH_PATH}`). Each web-checks every factual claim — cited works, legal cases, statute/regulation numbers + rules, professor titles/affiliations, named projects, dates — for existence and accuracy, classifying `OK / MISCHARACTERIZED / NOT_FOUND / WRONG_ATTRIBUTION|WRONG_DETAIL / NO_CITATION / UNCERTAIN`, with a source URL for every non-OK verdict and a strict no-guess rule. For batches > ~20, run this as the `find → adversarial-verify` Workflow above (a second independent agent re-checks each flag).
3. **Act on the gate (BLOCKING)** — every non-OK verdict must be fixed before drafts ship: swap a fabricated/mischaracterized citation for a verified work or de-specify to the research area; correct a wrong statute number / case / year / title against the source; delete a fabricated project. **"If you didn't find it, you don't write it."** A specific named/dated/"forthcoming" item no search can confirm is assumed fabricated and pulled. Re-verify after fixes.

**Hard rule**: no draft ships with an unverified specific factual claim. Run on the WHOLE batch, never a sample — these errors cluster wherever the drafter wanted a "specific hook" and lacked a real one, so sampling misses exactly the dangerous ones.

### Phase 2.6 — Style post-review (MANDATORY for batches > 30)

Even when each subagent followed Phases A-G, the corpus has consistent drift patterns that only appear at scale. Run these 5 sub-steps:

**R1 — Mechanical normalization (script, 5 min)**
`python3 scripts/normalize_drafts.py DIR --year 2027 --student-name "..." --role "..." --affiliation "..."`
Unifies subject template (`PhD inquiry (Fall YYYY) — <topic> — Name`) and signature block (3 lines: name / role / affiliation).

**R2 — Audit (script, 1 min)**
`python3 scripts/audit_drafts.py DIR`
Scans for: AI tells (hedging/inflation/throat-clearing), length outliers, opening duplicates, generic paragraph-3, PII leaks. Outputs structured report.

**R3 — User reviews + gives tone feedback**
Show user 3-5 representative drafts; ask for tone direction. Past lessons live in `references/style_lessons.md`.

**R4 — Bulk patch via subagents**
- Render patch prompts: `python3 scripts/render_patch_prompts.py DIR` (or inline via `references/style_patch_prompt.md`).
- Spawn batches of 7 Sonnet patch subagents over all 132 drafts. Each patches body only, preserving subject/signature/research.
- Re-render MDs: `python3 scripts/draft_io.py json-to-md DIR`
- Re-audit: confirm 0 em-dashes, ASK-element coverage >95%, no duplicates.
- Final regex cleanup pass for residual em-dashes the LLM missed (rare but happens).

**R5 — Sediment new lessons into `references/style_lessons.md`** (append-only). If any new pattern was caught, update the hard-rules section so next student's run avoids it from the start.

The 4 scripts and 1 prompt this Phase uses:

     ```markdown
     - 输出: [[个性化材料/PhD套磁/邮件草稿/INDEX|INDEX.md]]
     - 下一步: 老板逐封审核，附 RP+CV 后发出
     ```
   - If the advisor's 日报 file doesn't exist, create it with the header `# 日报-{顾问}` then add this entry.

### Phase 5 — Handoff message

Print a concise closing message:

- ✅ `{N}` 封草稿写好 (`{M}` 位已联系跳过)
- 📂 `INDEX.md` 在 `01_Student/{学生名}/个性化材料/PhD套磁/邮件草稿/`
- ✏️ 学生档案 + 待办看板 + 日报已更新 (顾问: `{中期顾问}`)
- ⚠️ 请逐封 review，附 RP + CV 再手动发出。回复后回来在 INDEX 里画掉对应行。

### Phase 6 — Reply tracking & progress sync (recurring, post-send)

Every time the user says "看看有没有回复" / "跟一下", run a **dated reconcile**, then **offer a WeChat summary**. This phase repeats over days/weeks.

1. **Sweep INBOX with real dates.** osascript over the bound account's INBOX (recent window), capture `extract address from sender` + `date received` (build a sortable `Y-M-D H:MM`) + subject. Cross-ref senders against the active manifest; **dedup against the tracker's already-handled set** (rows whose `last_contact` says 已回复). Distinguish "newly **arrived**" (by date) from "newly **handled**" — never call a 3-day-old reply "new" without its date (the user *will* catch it; 潘喆 2026-06-06).
2. **Split out bounces.** Also scan INBOX for `mailer-daemon` DSNs in the window and read each one's `Final-Recipient` + `Diagnostic-Code`. A bounced recipient **never received the mail** → move it out of "已发未回" into a `退信` status. **A bounce is usually signal**: web-verify the person's current role before any resend — repeatedly it's "left academia" (emeritus / law firm / industry policy), in which case there is no viable address and you must NOT guess one (guessing just re-bounces). Only resend when you find a *verified, currently-supervising* address.
3. **Classify & reply.** For genuine new replies: decline → gracious thanks (send); positive/scheduled → log the time window; substantive-feedback-on-the-RP → draft a reply but **HOLD it** for the student to confirm the research framing (it's *their* contribution to stake). No "I read X" overclaims; Dr. unless confirmed Professor.
4. **Send hygiene (Mail.app).** Before any reply-send, check **Outbox=0** — a single stuck message (e.g. an old practice email to a different student) silently jams `send_drafts_via_mailapp.py` (it reads Outbox>0 as "my send stuck" and halts). Clear it: `delete (first message of mailbox "Outbox")`. Gmail Drafts resist deletion when a **zombie compose window** is still open — `close (every window whose name contains "{subject}") saving no` first, *then* the id-collect delete sticks. Don't run a blanket "delete all drafts" while a legit unsent reply is still pending.
5. **Update the tracker** (`…套磁跟踪_数据源.json` → regenerate the Excel via `phd_supervisor_match/scripts/generate_tracker_excel.py`). Commit the vault.
6. **Offer the WeChat summary.** End the round with *"要不要我整理成一条微信消息发给 {学生}？"* If yes, produce it per [`references/wechat_progress_update.md`](references/wechat_progress_update.md) — honest headline numbers (replied / declined / bounced / silent, with the "feels like everyone replied" trap called out), positives tiered by heat with last-contact date+content (earlier rounds included), declines+unsuitable (funding blockers go here), one concrete action item. Save it alongside the tracker and commit.

## Shortlist format

**This is a FALLBACK format** — preferred path is the Excel autodiscovery (Phase 0 step 7). Only use this when neither upstream Excel exists for the student.

`01_Student/{学生名}/个性化材料/PhD套磁/导师名单.md` is a markdown file. The skill parses **any markdown table** with at least these columns (column header matching is case-insensitive, Chinese or English both fine):

| 必需 | 列名（可接受） |
|---|---|
| ✅ | `Supervisor Name` / `导师` / `姓名` |
| ✅ | `机构/学校` / `学校` / `University` / `School` |
| ⚪️ | `邮箱` / `Email` |
| ⚪️ | `总结` / `Summary` |
| ⚪️ | `重要论文` / `Papers` |
| ⚪️ | `重要链接` / `Links` / `URLs` |
| ⚪️ | `博士招生情况` / `Availability` |
| ⚪️ | `独特优势` / `Strength` |

Optional columns (⚪️) are passed to the research subagent as **preliminary info** to anchor identity disambiguation — they're not strictly required, but with them the research is much faster and less likely to find the wrong person of the same name.

Minimal example:

```markdown
# 导师名单

| Supervisor Name | 机构/学校 | 邮箱 |
|---|---|---|
| Wei Wang | National University of Singapore | wei.wang@nus.edu.sg |
| Yuki Tanaka | University of Tokyo | tanaka@u-tokyo.ac.jp |
```

Richer example (preferred, output of upstream `phd_supervisor_match` / `faculty-deep-dive` is a good source):

```markdown
| Supervisor Name | 机构/学校 | 邮箱 | 总结 | 重要论文 | 重要链接 |
|---|---|---|---|---|---|
| Wei Wang | NUS | wei.wang@nus.edu.sg | 主要研究人机交互... | Wang & Lee (2024) ... | https://nus.edu.sg/...\nhttps://scholar.google.com/... |
```

(`\n` inside cells = line break; pipe `|` inside cells must be escaped as `\|`.)

## Gotchas

- **🚨 Don't pin STUDENT's graduation year, only ADMISSION year**. Past audit on 132 drafts: Sonnet defaulted to "second-year MSc" / "finishing my MSc" in every draft, implying mid-YYYY graduation. For students whose graduation is uncertain (Chinese 学硕 may extend to year 3, etc.), this closes future options — if supervisor commits Fall YYYY slot but student graduates later, awkward retraction. **Fix**: subject + ASK keep `(Fall YYYY)` admission cycle, but body must NOT include student-tenure signals ("second-year", "finishing my MSc", "graduating in YYYY"). Run `references/tenure_neutral_patch_prompt.md` as standard post-pass when graduation timing is uncertain. Pre-flight check: ask user whether student's graduation date is firm before drafting.
- **🚨 Pattern-inferred emails are NEVER acceptable as confirmed**. Past audit on 132 supervisors: out of 24 "pattern-plausible" emails (where Sonnet guessed `firstname.lastname@uni.edu` based on institutional convention but no page directly showed the address), **3 (~12.5%) turned out wrong**: Marenduzzo (`Davide.Marenduzzo@ed.ac.uk` → real `dmarendu@ph.ed.ac.uk`); Yethiraj (`yethiraj@chem.wisc.edu` → real `yethiraj@wisc.edu`); 胡金莲 (`mejlhu@cityu.edu.hk` → real `jinlianu@cityu.edu.hk`). All caught only by the Phase 2.5 paper-PDF deep hunt. **DO NOT skip Phase 2.5.**
- **🚨 PII-redaction trap** — Subagent Write tool calls can silently turn real emails like `tfm@caltech.edu` into the literal placeholder string `[email protected]` (markdown autolink display form). 3/132 supervisors hit this trap. The verifier catches it (the literal `[email protected]` string can't ever be in a fetched page so it always fails grep). When you spawn the re-hunt, tell the subagent explicitly: "Write the literal email characters to JSON; never write `[email protected]` placeholder."
- **PDF reading** — the Read tool supports PDF natively (per its description). For RP/CV PDFs, just `Read` the path; no need for `pdftotext` shell-outs.
- **TeX CV** — if the CV is a `.tex` file, Read it as plain text. Strip `\highlight{...}` wrappers conceptually (the n8n equivalent gets the rendered text via pdfextract, so we want similar — but for prompt purposes the raw LaTeX is fine; Claude can read past it).
- **`套磁过了` annotations in Excel** — `phd_supervisor_match` Excel has a `匹配原因` column that the user manually annotates with `套磁过了` / `已联系` / `已套磁` / `已发出` / `已发邮件` to track outreach. The parser detects these and marks `skip=true`. Phase 0 confirms with the user before excluding them — the user can override per-row if they want to RE-send (rare, but possible after waitlist follow-up).
- **Merged cells in Excel** — `phd_supervisor_match` has 3 levels of merged cells (地区 / 学校 / 院系/方向). `faculty-deep-dive` merges 4 columns across paper rows. The parser forward-fills these correctly — do NOT try to read Excel with naive `openpyxl` iteration in another script without forward-fill; you'll get blank rows.
- **Supervisor name with non-ASCII chars** — when writing the output file, sanitize for filesystem (replace `/` `\` `:` `*` `?` `"` `<` `>` `|` with `_`). Keep the original name in the frontmatter + body.
- **`英文名` for Excel discovery** — the parser glob `*Supervisor_Shortlist_*.xlsx` matches by any-prefix, so it works even if you don't know the student's English name. But the YAML `英文名` is still worth reading for the file-naming Phase 4 outputs.
- **`中期顾问` is multi-value capable** — vault YAML can have `中期顾问: 王世杰` OR `中期顾问: [王世杰, 陆梦婕]`. Phase 4 日报 step MUST write to all advisors' files when it's a list. See the 2026-05-06 ultrareview lesson in vault `CLAUDE.md`.
- **No write to vault git index** — vault is `git init`'d but commits are at the user's discretion. Don't `git add` or `git commit` inside the vault — just write files.
- **WebSearch / WebFetch tool availability** — if they're deferred at runtime, the subagent will need to load them first via ToolSearch. Mention this in the subagent prompt so it doesn't get stuck.
- **Don't conflate with `phd_supervisor_match` or `faculty-deep-dive`** — those are vault skills that PRODUCE the shortlist. This skill CONSUMES a shortlist. If the user has no shortlist yet, point them at those vault skills first (run inline per the sdg-html `CLAUDE.md` Pattern B), don't try to build a shortlist from scratch in this skill.
